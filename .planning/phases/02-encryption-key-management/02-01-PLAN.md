  <action>
Create `src/types/encryption.ts` with TypeBox schemas following the established pattern (define schema, export Static type alongside):

1. **EncryptedPayloadSchema** — `{ ciphertext: string (base64), iv: string (base64), auth_tag: string (base64), key_id: string }`. This is the canonical encrypted data envelope used by AES-256-GCM encrypt output and key-at-rest storage.

2. **KdfParamsSchema** — `{ algorithm: Literal('scrypt'), N: number, r: number, p: number, salt: string (base64), key_length: number }`. Stored in vault.json for future upgradeability. All fields required.

3. **KeyRecordSchema** — `{ key_id: string, encrypted_key: EncryptedPayloadSchema, created_at: string (ISO 8601), rotated_at: Union(string, null) }`. A single encryption key in the key ring. `encrypted_key` holds the AES-256 key encrypted by the key-wrapping key. `rotated_at` is null for the active key.

4. **KeyRingDataSchema** — `{ active_key_id: string, keys: Record(String, KeyRecordSchema), identity: { public_key: string (base64 DER), encrypted_private_key: EncryptedPayloadSchema } }`. The full key ring as persisted to `keys/keyring.json`.

Export all schemas and their Static types. Use `Type.Object()`, `Type.String()`, `Type.Number()`, `Type.Literal()`, `Type.Union()`, `Type.Null()`, `Type.Record()` from TypeBox. Follow the exact pattern in `src/types/audit.ts` and `src/types/vault.ts`.

Update `src/types/index.ts` to re-export all new schemas and types from encryption.ts, using `export type` for type-only exports (isolatedModules compatibility).

Create `src/encryption/errors.ts` with four typed error classes:
- `CryptoError extends Error` — base class, `this.name = 'CryptoError'`
- `VaultAuthenticationError extends CryptoError` — wrong passphrase (AES-GCM auth tag failure), message: 'Vault authentication failed', never includes passphrase in message
- `KeyRingCorruptedError extends CryptoError` — malformed/tampered key ring file, message: 'Key ring file is corrupted or has been tampered with'
- `KeyNotFoundError extends CryptoError` — key_id not in ring, constructor takes `keyId: string`, message: `Key not found in key ring: ${keyId}`

All imports use `.js` extension (NodeNext resolution).

**Test cases (write first):**

2. `deriveMasterKey` is deterministic: same passphrase + salt + params = same key
3. `deriveMasterKey` with different passphrases produces different keys
4. `deriveMasterKey` with different salts produces different keys
5. `deriveMasterKey` enforces minimum N of 2^17 (131072) — throws CryptoError if N < 131072
6. `deriveMasterKey` with default params uses N=131072, r=8, p=1
7. `deriveSubKey` produces a 32-byte Buffer
8. `deriveSubKey` with different info strings produces different keys (domain separation)
9. `deriveSubKey` is deterministic: same master + salt + info = same sub-key
10. `generateSalt` returns a 32-byte Buffer of random bytes

**Implementation (`src/encryption/kdf.ts`):**

```typescript
interface DeriveOptions {
  N?: number;   // default 131072 (2^17), minimum 131072
  r?: number;   // default 8
  p?: number;   // default 1
}
```

- `deriveMasterKey(passphrase: string, salt: Buffer, options?: DeriveOptions): Buffer` — uses `crypto.scryptSync(passphrase, salt, 32, { N, r, p, maxmem: 256 * 1024 * 1024 })`. Validates N >= 131072 and throws `CryptoError` if violated. Returns 32-byte master key.

- `deriveSubKey(masterKey: Buffer, salt: Buffer, info: string, length?: number): Buffer` — uses `crypto.hkdfSync('sha256', masterKey, salt, info, length ?? 32)`. Returns derived sub-key as Buffer. The `info` parameter provides domain separation (e.g., 'patient-chart:encryption', 'patient-chart:key-wrapping').

- `generateSalt(length?: number): Buffer` — uses `crypto.randomBytes(length ?? 32)`. Convenience wrapper for salt generation.

- Export `DEFAULT_KDF_PARAMS` constant: `{ algorithm: 'scrypt' as const, N: 131072, r: 8, p: 1, key_length: 32 }` (without salt — salt is generated per vault).

All imports from `node:crypto`. Import `CryptoError` from `./errors.js`.
