    TDD RED-GREEN-REFACTOR for Ed25519 signing and verification.

    **Function signatures:**
    - `generateEd25519KeyPair(): { publicKey: KeyObject, privateKey: KeyObject }`
    - `sign(data: Buffer, privateKey: KeyObject): Buffer` — returns 64-byte signature
    - `verifySignature(data: Buffer, signature: Buffer, publicKey: KeyObject): boolean`
    - `exportEd25519PublicKey(key: KeyObject): Buffer` — DER SPKI (44 bytes)
    - `exportEd25519PrivateKey(key: KeyObject): Buffer` — DER PKCS8 (48 bytes)
    - `importEd25519PublicKey(der: Buffer): KeyObject`
    - `importEd25519PrivateKey(der: Buffer): KeyObject`

    **Test cases (RED phase):**
    1. generateEd25519KeyPair returns publicKey and privateKey KeyObjects
    2. sign returns a 64-byte Buffer
    3. sign + verifySignature round-trip returns true on original data
    4. verifySignature returns false on tampered data
    5. verifySignature returns false with wrong public key
    6. sign uses null algorithm (Ed25519 internal SHA-512) — no algorithm parameter
    7. exportEd25519PublicKey returns 44-byte DER Buffer
    8. exportEd25519PrivateKey returns 48-byte DER Buffer
    9. export + import round-trip: sign with original, verify with reimported — succeeds
    10. DER public key starts with Ed25519 SPKI header (302a300506032b6570032100)
    11. sign handles large data (1MB) without error

    **Implementation (GREEN phase):**
    - Use `crypto.generateKeyPairSync('ed25519')` for key generation.
    - Use `crypto.sign(null, data, privateKey)` — algorithm is null for Ed25519 (RFC 8032 EdDSA uses internal SHA-512).
    - Use `crypto.verify(null, data, publicKey, signature)`.
    - DER export: `key.export({ type: 'spki', format: 'der' })` for public, `key.export({ type: 'pkcs8', format: 'der' })` for private.
    - DER import: `createPublicKey({ key: der, format: 'der', type: 'spki' })` for public, `createPrivateKey({ key: der, format: 'der', type: 'pkcs8' })` for private.
    - Define DER header constants: `ED25519_SPKI_HEADER` and `ED25519_PKCS8_HEADER` as readonly Buffers for documentation/validation.

    **Constraints:**
    All crypto via node:crypto only. Ed25519 algorithm parameter is ALWAYS null.
    Do NOT name the sign function `ed25519Sign` — use `sign` since it's in the ed25519 module namespace. Consumers import as `import { sign } from './encryption/ed25519.js'`.
    DER headers are constants defined at module level for clarity and potential validation.

    - `generateX25519KeyPair(): { publicKey: KeyObject, privateKey: KeyObject }`
    - `computeSharedSecret(ownPrivateKey: KeyObject, peerPublicKey: KeyObject): Buffer` — returns 32-byte shared secret
    - `exportX25519PublicKey(key: KeyObject): Buffer` — DER SPKI (44 bytes)

    - `importX25519PublicKey(der: Buffer): KeyObject`

    **Test cases (RED phase):**
    1. generateX25519KeyPair returns publicKey and privateKey KeyObjects
    2. computeSharedSecret returns a 32-byte Buffer
    3. Both sides produce identical shared secret (alice computes with bob's public, bob computes with alice's public — must be equal)
    4. Different key pairs produce different shared secrets
    5. Low-order public key (all-zeros 32 bytes) is rejected with an error
    6. exportX25519PublicKey returns 44-byte DER Buffer
    7. exportX25519PrivateKey returns 48-byte DER Buffer
    8. export + import round-trip: compute shared secret with reimported keys — same result
    9. DER public key starts with X25519 SPKI header (302a300506032b656e032100)
    10. computeSharedSecret wraps OpenSSL errors in CryptoError

    **Implementation (GREEN phase):**
    - Use `crypto.generateKeyPairSync('x25519')` for key generation.
    - Use `crypto.diffieHellman({ publicKey: peerPublicKey, privateKey: ownPrivateKey })` for shared secret computation.
    - DER export/import follows the same pattern as Ed25519 but with X25519 OID headers.
    - OpenSSL automatically rejects low-order X25519 public keys (all-zeros, order-2, order-4, order-8) with `ERR_OSSL_PROVIDER_FAILED_DURING_DERIVATION`. Wrap in CryptoError.
    - Define `X25519_SPKI_HEADER` and `X25519_PKCS8_HEADER` as readonly Buffer constants.

    **Constraints:**
    All crypto via node:crypto only. Low-order point rejection is automatic via OpenSSL — no custom validation needed.
    Shared secret is exactly 32 bytes, suitable as an AES-256-GCM key directly.
    Wrap all node:crypto errors in CryptoError for consistent error surface.
  </action>
