  <name>Task 1: Knowledge TypeBox schemas and error classes</name>
  <files>src/types/knowledge.ts, src/knowledge/errors.ts, src/knowledge/schema.ts</files>
  <action>
Create three new files:

**src/types/knowledge.ts** — TypeBox schemas for knowledge note frontmatter metadata, following the exact pattern used in `src/types/encryption.ts` (import Type + Static from @sinclair/typebox, define schema, export schema + Static type).

Define these schemas:
1. `ClinicalStatusSchema` — TypeBox Union of 5 literals: 'active', 'inactive', 'resolved', 'recurrence', 'remission' (FHIR R4 Condition.clinicalStatus). Export type `ClinicalStatus`.
2. `VerificationStatusSchema` — TypeBox Union of 5 literals: 'confirmed', 'provisional', 'differential', 'refuted', 'entered-in-error' (FHIR R4 Condition.verificationStatus). Export type `VerificationStatus`.
3. `NoteTypeSchema` — TypeBox Union of 11 literals: 'condition', 'medication', 'allergy', 'lab', 'imaging', 'procedure', 'provider', 'encounter', 'directive', 'document', 'problem_list'. Export type `NoteType`.
4. `KnowledgeNoteMetaSchema` — TypeBox Object with fields:
   - `id`: String (UUIDv7 note identifier)
   - `type`: NoteTypeSchema
   - `status`: Union of 'active', 'inactive', 'resolved' (lifecycle status of note itself)
   - `created`: String (ISO 8601)
   - `updated`: String (ISO 8601)
   - `source_entries`: Optional Array of Strings (ledger://entry/<id> provenance links)
   - `tags`: Optional Array of Strings
   - `clinical_status`: Optional ClinicalStatusSchema (condition-specific)
   - `verification_status`: Optional VerificationStatusSchema (condition-specific)
   - `snomed_ct`: Optional String (SNOMED CT concept ID, numeric string)
   - `snomed_display`: Optional String (SNOMED CT fully specified name)
   - `icd10`: Optional String (ICD-10-CM code, e.g. "E11")
   - `icd10_display`: Optional String (ICD-10-CM display name)
   - `onset`: Optional String (ISO 8601 date of condition onset)
   - `chronic`: Optional Boolean
   Export type `KnowledgeNoteMeta`.

Include JSDoc comments on each schema explaining its clinical purpose (see research for context).

**src/knowledge/errors.ts** — Error classes following the pattern from `src/encryption/errors.ts` (simple inheritance with descriptive `this.name`):
1. `KnowledgeStoreError` extends Error — base error for all knowledge operations. Constructor takes `message: string`.
2. `NoteNotFoundError` extends KnowledgeStoreError — thrown when reading a note that doesn't exist. Constructor takes `relativePath: string`, message: `Knowledge note not found: ${relativePath}`.
3. `NoteCorruptedError` extends KnowledgeStoreError — thrown when a note file fails EncryptedPayload schema validation or decryption. Constructor takes `relativePath: string`, message: `Knowledge note corrupted or tampered with: ${relativePath}`.
4. `PathTraversalError` extends KnowledgeStoreError — thrown when relativePath resolves outside knowledge/ directory. Constructor takes `relativePath: string`, message: `Path traversal detected: ${relativePath}`.

**src/knowledge/schema.ts** — Constants and re-exports:
1. `KNOWLEDGE_SUBDIRS` — const array of the 10 clinical domain subdirectory names: `['conditions', 'medications', 'allergies', 'labs', 'imaging', 'procedures', 'providers', 'encounters', 'directives', 'documents']` as const.
2. Export type `KnowledgeSubdir` (typeof KNOWLEDGE_SUBDIRS[number]).
3. Re-export `KnowledgeNoteMetaSchema` and type `KnowledgeNoteMeta` from `../types/knowledge.js`.

All imports must use `.js` extension (NodeNext resolution).
